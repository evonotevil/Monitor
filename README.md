# 🌐 Lilith Legal · 全球游戏合规动态监控

> 面向合规团队的自动化情报工具，每天自动追踪全球主要市场的游戏监管动态，翻译整理后推送到飞书，并沉淀为可查阅的周报。

---

## 这个工具能帮你做什么

你不需要每天手动搜索各国监管机构的网站或新闻。这套系统会自动完成以下工作：

1. **每天早上 8 点左右**，通过官方 RSS、Google News 和 GDELT 补充源抓取最近日报自然日窗口内的全球游戏合规动态
2. **12 语种统一监控**，覆盖英语、日语、韩语、越南语、印尼语、泰语、繁体中文、葡萄牙语、西班牙语、德语、法语和阿拉伯语
3. **四通道检索**，每种语言同时覆盖监管合规、执法诉讼、平台政策和重点游戏公司/产品
4. **AI 自动翻译与两级地理分类**，将多语言原文整理成规范的中文标题和摘要，同时记录一级区域、具体管辖区和适用范围
5. **AI 风险评估**，从营收影响、产品改动、时间紧迫性、影响范围四个维度综合评分，帮助团队快速判断优先级
6. **追踪民事诉讼**，覆盖集体诉讼、产品责任诉讼、成瘾设计索赔等影响行业的重大案件
7. **监控游戏存续立法**，追踪在线服务停运、数字所有权、游戏保存等新兴消费者权益议题（如加州 AB 1921 Protect Our Games Act）
8. **行业公司动态追踪**，内置全球 50+ 家主要游戏发行商/开发商名单（覆盖中资出海、日韩、欧美、东南亚头部公司），自动捕获涉及这些公司的执法、诉讼、处罚和合规动态
9. **多层过滤与跨语言去重**，排除博彩、体育、硬件评测、公司财报等噪音，并合并同一事件的多语种报道
10. **工作日推送飞书日报**，按地区分类展示当日最重要的合规动态
11. **写入飞书多维表格**，供团队人工标注和筛选，含四维风险评分和信源等级
12. **每周手动发布周报**，生成上一个完整自然周（周一至周日）的手机端/电脑端 HTML 报告和 PDF

## 日常使用流程

### 每天（自动，无需操作）

```
早上 8:00 自动运行
    ↓
RSS → 12 语种 Google News 四通道 → GDELT 少量兜底
    ↓
标题去重 → 游戏/监管双信号过滤 → 原文日期校正并再次过滤
    ↓
AI 翻译分类 → 一级区域/具体管辖区/适用范围归类 → 四维风险评估 → 跨语言事件去重
    ↓
写入飞书多维表格（每天，含周末）
    ↓
推送飞书日报卡片（仅周一至周五）
```

**你收到飞书日报后可以做的事：**
- 查看当日有哪些地区有新动态
- 点击标题链接查看原文
- 在飞书多维表格里对每条动态标注处理状态（见下方说明）

**运行状态说明：**
- 抓取成功但没有新动态：工作日发送绿色「昨日无新增合规动态」卡片
- 抓取步骤失败：发送红色「监控任务异常」卡片，跳过多维表格同步和正常日报，同时将 GitHub Actions 标记为失败
- GDELT 作为补充源，日报首次遇到 429 会立即停止；GDELT 失败不会影响 RSS 和 Google News 主流程
- 周一自动覆盖周六、周日和周一，其他工作日覆盖今天和昨天的自然日期窗口

---

### 每周（手动触发）

团队完成多维表格初筛后，在 GitHub 上手动触发「发布周报」。系统会：

1. 从飞书多维表格读取上一个完整自然周（周一至周日）内已审核的条目
2. 按处理状态自动分为三个区块生成报告：
   - 🎉 **上周已完成任务**：状态为「✅ 已合规/归档」的条目，展示跟进 BP、核心结论和文档链接
   - 🌏 **上周全球合规动态汇总**：状态为「📰 行业动态」的条目，按国家/地区分类展示
   - 🎯 **本周跟进任务**：状态为「🏃 处理/跟进中」的条目，按跟进 BP 分组展示各自名下的待办条数
3. 同时生成手机端和电脑端两份 HTML 报告 + PDF 版本
4. 推送飞书周报卡片（含报告直链）

## 飞书多维表格 — 团队工作台

多维表格是团队日常工作的核心入口。每天新抓取的条目会自动写入，初始状态为「🤖 待初筛」。

### 处理状态说明

| 状态 | 含义 | 周报中的位置 |
|------|------|-------------|
| 🤖 待初筛 | 系统自动写入，尚未人工确认 | ❌ 不纳入周报 |
| 📰 行业动态 | 确认为有价值的行业/监管动态 | 「上周全球合规动态汇总」区块 |
| 👤 待研判 | 需要 BP 进一步研判跟进 | ❌ 不纳入周报（研判完成后改为其他状态） |
| 🏃 处理/跟进中 | BP 正在跟进处理 | 「本周跟进任务」区块（**不受发布日期限制**，持续跟进的老任务始终显示） |
| ✅ 已合规/归档 | 已处理完毕并归档 | 「上周已完成任务」区块 |
| 🗑️ 噪音/不推送 | 判断为无关或低价值 | ❌ 不纳入周报 |

> **重要**：只有状态不是「🤖 待初筛」和「🗑️ 噪音/不推送」的条目，才会被纳入周报。所以每周发布前，请确保团队已完成初筛。

### 主要字段说明

| 字段 | 说明 |
|------|------|
| 动态标题 | AI 生成的中文标题，格式为 [国家/地区] 事件描述 |
| 摘要 | AI 生成的 60-90 字合规摘要，说明具体规则和需要采取的行动 |
| 💡 核心结论 | 人工补充的法务判断（若填写，周报优先展示此字段而非摘要） |
| 国家/地区 | 按大区分组：北美 / 欧洲 / 日韩 / 港澳台 / 东南亚 / 中东 / 南美 / 大洋洲 / 其他 |
| 具体国家/地区 | 具体法律管辖区，例如美国、法国、台湾地区、新西兰或欧盟；全球、多国和未确定动态留空 |
| 适用范围 | 单一管辖区 / 超国家管辖区 / 多国/地区 / 全球 / 未确定 |
| 合规类别 | 见下方分类说明 |
| 处理状态 | 见上方状态说明 |
| 跟进 BP | 负责跟进的合规 BP（周报归档区和跟进区展示） |
| 协助 BP | 协同支持的 BP（周报中拼接显示在跟进 BP 后） |
| 法务结论 | 法务 BP 填写的结论（周报归档区展示） |
| 专项合规文档 | 超链接字段（周报归档区展示文档按钮） |
| 归档日期 | 条目归档时间，由飞书自动化写入（归档条目按此日期纳入上一个完整自然周；跟进中条目不受此限制） |
| 发布日期 | 原始文章发布日期（行业动态按此日期纳入上一个完整自然周） |
| 原始链接 | 点击可查看原文 |
| 信源名称 | 来源媒体或机构名称 |
| 营收影响 | AI 评估的营收风险等级（0-3），0=无影响，3=下架/大额罚款 |
| 产品改动 | AI 评估的产品改动量（0-3），0=无需改动，3=核心玩法重构 |
| 时间紧迫性 | AI 评估的时间紧迫度（0-3），0=无时间要求，3=已生效/<1个月 |
| 影响范围 | AI 评估的影响范围（0-3），0=单一非核心市场，3=全球 |

> **风险评分字段**：需在多维表格中手动添加「营收影响」「产品改动」「时间紧迫性」「影响范围」四个数字列。未添加时系统正常运行，只是不写入这些字段。

> **两级地理字段**：建议在多维表格中增加单选字段「具体国家/地区」和「适用范围」。程序写入前会读取真实字段结构；字段尚未建立或结构读取失败时只写原有字段，不会让整批同步失败。

> **覆盖说明**：北美、欧洲等核心市场以官方 RSS、法律情报和英语查询为主；东南亚、日韩、港澳台、中东和拉美通过当地语言四通道查询补强。GDELT 仅保留三条全球复合查询作为少量兜底，不再按国家逐条请求。部分长尾国家仍依赖新闻索引覆盖，稳定性低于核心市场。

### 多语种与地区判定

- 每个支持语种固定使用四类查询：监管合规、执法诉讼、平台政策、重点公司/产品
- 查询词和相关性过滤词由同一份语言画像维护，避免出现“已经抓到、但过滤器不认识当地执法词”的情况
- `国家/地区`继续保存九大一级区域；`具体国家/地区`保存标准化国家、地区或欧盟
- 文章语言不等于事件地区。例如日语媒体报道 FTC 执法，会归为「北美 / 美国」，而不是日本
- 判断优先级为监管机构/法院/法规、正文明确国家、官方来源提示、LLM，最后才使用 Google News locale 兜底
- 欧盟是超国家法律管辖区，可以写入「具体国家/地区」；全球、多国/地区和未识别不是管辖区，只写入「适用范围」
- 多国动态只保存一个主要管辖区；没有明确主要管辖区时具体管辖区留空，不强行猜测
- 历史数据仅按与现有一级区域一致的明确国家、监管机构、法律或标题前缀回填；发现冲突时保持为空
- LLM 分类后再次执行事件级去重。事件指纹只负责发现候选，必须由 LLM 确认后才合并；LLM 不可用时保守保留

## 日报卡片怎么看

飞书日报卡片会在每个工作日早上 8 点左右收到。无新增动态时会推送绿色卡片；抓取失败时会推送红色异常卡片，明确提示不能将本次结果理解为「无新增」。周一的日报会自动补上周六和周日的动态。

**卡片结构：**
- **顶部**：新增条数（周一显示「周末至今新增 N 条」），以及各地区动态数量概览
- **AI 综述**（灰色引用块）：150 字以内的整体情况总结
- **按地区分块**：每个有动态的地区单独一栏，最多展示 3 条，全局最多展示 12 条；若该地区实际条数超过展示数，标题会显示「昨日 N 条（展示 M 条）」
- **每条动态包含**：风险等级（🔴高 / 🟠中 / 🔵一般）、一级区域与具体管辖区、状态、合规类别、标题链接、机制变动摘要

**去重机制：** 同一事件被多家媒体或不同语种报道时，系统综合 URL、中文标题相似度、事件指纹和 LLM 判断，只展示信源最权威的那条（官方机构 > 法律情报 > 行业媒体 > 普通新闻）。事件指纹不会单独触发删除

**卡片颜色含义：**
- 🟢 绿色 Header：无新增动态
- 🔴 红色 Header：正常日报中代表最高风险动态（影响评分 ≥ 9）；标题为「监控任务异常」时代表抓取失败
- 🟠 橙色 Header：当日有高风险动态（影响评分 ≥ 7）
- 🔵 蓝色 Header：一般重要性动态

## 周报卡片怎么看

手动触发周报发布后，飞书群会收到一张周报卡片。

**卡片结构：**
- **统计概览**：顶部固定一行展示三区数据——归档条数、行业动态条数、跟进任务条数，一目了然
- **地区标签**：归档条目涉及的地区（按大区分组）
- **AI 摘要**（灰色引用块）：上周关键词 + 建议关注事项
- **跟进任务**（有跟进时展示）：按 BP 分组展示各自名下的待办条数
- **按钮行**：查看周报（手机端）、PC 版周报、查看多维表格，三个按钮并排

## 风险评估机制

系统采用 **AI 四维评估 + 正则兜底** 的双轨评分机制，帮助团队快速判断每条动态的优先级。

### 四个风险维度（每项 0-3 分）

| 维度 | 0 | 1 | 2 | 3 |
|------|---|---|---|---|
| **营收影响** | 无直接影响 | 间接影响 | 罚款/部分受限 | 下架/大额罚款/支付被封 |
| **产品改动** | 无需改动 | 轻微（政策文案） | 中等（年龄验证/概率公示） | 重大（核心玩法重构） |
| **时间紧迫性** | 无时间要求 | >6 个月 | 1-6 个月 | 已生效或 <1 个月 |
| **影响范围** | 单一非核心市场 | 单一核心市场 | 多核心市场/全区域 | 全球/主要平台全局策略 |

### 综合评分

四个维度通过加权公式映射到 **1.0-10.0 分**（≥9 高风险 / ≥7 中风险 / ≥5 关注 / <5 低优先）。

AI 评估搭载在翻译+分类的同一次 LLM 调用中，不产生额外 API 费用。当 LLM 不可用时，自动退回基于关键词匹配的正则评分。

## 常见问题

**Q：今天没收到日报，是系统出问题了吗？**
A：工作日每天都会推送日报。无新增时发送绿色卡片，抓取失败时发送红色异常卡片；如果完全没收到，可以去 GitHub Actions 页面查看当日运行日志。周末不推送飞书卡片，但数据照常抓取，周一日报会自动补上周末的动态。

**Q：红色卡片是否一定代表高风险新闻？**
A：不一定。正常日报的红色 Header 表示存在影响评分 ≥9 的高风险动态；如果卡片标题是「全球游戏合规监控异常」，则表示抓取任务失败，本次不能判断是否存在新动态。

**Q：多维表格里的条目看起来不相关，是 AI 判断错了吗？**
A：AI 有一定误判率。遇到这种情况，把处理状态改为「🗑️ 噪音/不推送」即可，这条内容就不会进入周报。如果某类噪音频繁出现，可以告知技术负责人添加过滤规则。系统对官方/法律信源已增加数字行业信号词过滤，会自动排除传统行业（食品、通信贩卖等）的监管新闻。

**Q：我在多维表格里填写了「💡 核心结论」，周报里会显示吗？**
A：会。如果填写了核心结论，周报里这条动态的摘要部分会优先显示核心结论的内容，而不是 AI 自动生成的摘要。

**Q：周报是自动发送的吗？**
A：否。周报需要团队在完成多维表格初筛后，手动在 GitHub Actions 页面点击「Run workflow」触发发布。系统会按触发时点自动生成上一个完整自然周（周一至周日）的报告。

**Q：系统能追踪游戏行业的民事诉讼吗？**
A：可以。系统会自动监控集体诉讼（class action）、产品责任诉讼（product liability）、成瘾设计索赔等影响行业的重大案件。这些诉讼虽非政府监管行为，但可能导致天价和解金、应用商店下架或推动后续立法，属于合规风险前哨信号。

**Q：系统能追踪游戏停服/数字所有权相关的立法吗？**
A：可以。系统会自动监控游戏存续（game preservation）、在线服务停运（live service shutdown）、数字所有权（digital ownership）等议题，包括 Stop Killing Games 运动和加州 AB 1921《Protect Our Games Act》等立法动态。这类法案要求发行商在停服时提供离线补丁或全额退款，可能影响在线服务型游戏的运营策略。

**Q：系统能追踪其他游戏公司的合规动态吗？**
A：可以。系统内置了全球 50+ 家主要游戏发行商和开发商的名单，覆盖中资出海、日韩、欧美、东南亚等地区的头部公司。当这些公司涉及执法、诉讼、处罚或合规事件时，即使新闻标题不含"game"等通用关键词，系统也会自动捕获。公司名单定期更新，如需增补可联系技术负责人。

**Q：我想查某个关键词的历史动态，怎么做？**
A：可以直接在飞书多维表格里筛选，也可以联系技术负责人通过数据库查询。

### 质量控制与验证

- 日报抓取日志会输出各语种在 `raw → title_unique → rule_relevant → date_rechecked → classified` 各阶段的数量
- Apple Developer News、Android Developers Blog 等日期不可信来源会抓取原文日期，并在校正后重新执行时间窗口过滤
- 已移除长期返回损坏 XML/HTML 的 EUR-Lex RSS、GDPRHub Atom、JD Supra Consumer Protection RSS 和 Brazil ANPD RSS；相关覆盖由现有有效 RSS、当地语言查询和官方站点查询承担
- GitHub Actions 通过内部变量 `FETCH_STEP_OUTCOME` 将抓取结果传给日报；失败时发送红色卡片并以非零状态结束
- 修改抓取、分类或日报逻辑后，运行 `python -m pytest tests/ -q`；本次更新后共 `392 passed`，覆盖工作流失败告警、自然日窗口、多语种过滤、GDELT 限速、两级管辖区、SQLite 迁移、飞书字段兼容和跨语言去重
